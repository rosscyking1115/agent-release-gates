# Model Card

## Project

Agent Release Safety Gates

## Intended Use

This project provides a public evaluation harness for local experimentation with AI-agent safety and reliability:

- grounded retrieval over synthetic runbooks
- external public technical-support retrieval evaluation using a 480-case TechQA-RAG-Eval sample
- external public enterprise-support retrieval evaluation using a 160-case WixQA expert-written sample
- deterministic release gates for benchmark, safety, governance, and observability checks
- structured extraction from synthetic tickets
- role-aware answer generation
- policy refusal for risky or weak-evidence requests
- approval-gated mock tool use
- traceable audit events, monitoring snapshots, OpenTelemetry-style span exports, a local trace timeline, and a queryable trace index

It is not intended for real banking, real customer support, real employee operations, or real workflow automation.

The controlled synthetic operations domain is the benchmark scenario. It should not be interpreted as a clone, critique, or measurement of any real organization's internal AI system.

## Data

Controlled operations data is generated by the local synthetic data generator. The TechQA and WixQA public RAG tracks are external public data and are reported separately from the synthetic benchmark.

## Known Limitations

- The current embedding-store implementation uses deterministic feature hashing, not a provider-backed embedding model or vector database.
- Ticket text and runbook text are synthetic and still partly templated, although the golden set now includes noisy abbreviations, missing metadata, false leads, typos, human-like phrasing, human email threads, manually authored chat/meeting/screenshot/handoff/control-room/redacted/stale-thread fragments, evidence packets, mixed manual review bundles, field notes, retrieved-context review packets, decision logs, partial evidence notes, analyst scratch notes, CSV excerpts, incident timelines, retrieved-document injection, adversarial instructions, conflicting evidence, and long conflicting context.
- The TechQA and WixQA public tracks currently use compact samples rather than the full upstream public datasets.
- A reviewed credentialed provider-backed embedding run (OpenAI `text-embedding-3-small`) is published for the synthetic benchmark in `reports/provider_embedding_published_result.json`. It matches the best local retrievers at 100% retrieval hit rate@3 and 100% citation coverage, so the controlled benchmark is saturated and does not discriminate provider from local retrieval. A reviewed provider-backed embedding result on the public RAG tracks is also published in `reports/public_rag_provider_embedding_published_result.json`: on the harder public data the hosted OpenAI embedding beats the local TF-IDF retriever on retrieval hit rate@3 (TechQA 89.06% vs 80.73%; WixQA 98.12% vs 77.50%), with matching gains in top-1 citation accuracy (TechQA 81.51% vs 72.40%; WixQA 84.38% vs 61.25%) and MRR@3. A multi-model comparison is not published yet.
- The local TF-IDF vector and local embedding-store retrievers pass the expanded synthetic golden suite. The hybrid sparse semantic retriever now has one manual field-note miss, which is kept in the report as a useful final-selection weakness rather than hidden by rewriting the benchmark.
- Structured extraction is deterministic pattern matching, not LLM extraction.
- Security checks and severity weights are deterministic controls, even after adding harder retrieved-context attacks and residual-risk scoring; they are not a full adversarial red-team harness.
- Safety evaluation includes sampled prevalence estimation, classifier threshold tuning, human-review queue simulation, reviewer-disagreement slicing, secondary review-band analysis, secondary-floor validation, mitigation-impact reporting, and a completed threshold decision memo. Reviewer behavior is still deterministic and synthetic.
- Real independent human labels, inter-rater agreement, and multi-model result tables are planned but not yet published. The external-review packet is prepared, and hosted LLM-as-judge evidence currently consists of one reviewed OpenAI calibration run.
- The controlled agent is a local workflow, not a LangGraph state machine yet.
- Telemetry export is local JSONL with a deterministic trace index and optional OTLP/HTTP collector adapter; CI verifies posting against a local capture endpoint and a Dockerized OpenTelemetry Collector metrics-verified deployment, but not downstream storage or production visualization.
- Metrics should be interpreted as engineering checks over synthetic cases, not claims about real-world production performance.

## Safety Boundaries

The system should refuse or abstain when:

- evidence is insufficient
- a prompt attempts to override policy
- a user asks for sensitive data
- a user asks to reveal system prompts or hidden instructions
- a user asks for unapproved side effects
- retrieved content appears to contain unsafe instructions

Side-effecting operations are mock-only and require explicit approval.
